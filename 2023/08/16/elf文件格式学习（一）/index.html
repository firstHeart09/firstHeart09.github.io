第一章 什么文件可以使用elf文件

elf文件可以使用三种文件格式，分别是目标文件、可执行文件以及共享文件

  • 目标文件（可重定位目标文件）通常是.o文件，包含数据和二进制代码，被链接器读取，用来产生一个可执行文件或共享文件
  • 可执行（目标）文件：该文件是链接后形成的可执行文件，即静态链接的可执行文件。包含二进制代码和数据，其形式可以被直接复制到内存并执行
  • 共享（目标）文件：通常是.so动态库文件或者动态链接形成的可执行（目标）文件，可以在加载或者运行时被动态的加载进内存并链接

第二章 elf文件格式


如上图所示，elf文件结构主要可以分为四个部分，分别是elf文件头、程序头表、段（节）、段头表。但并不是每一个elf文件都有程序头表和节头表这两张表。

对于elf文件格式的解析分为两种视图：一是链接视图、二是执行视图

  • ELF header： 描述整个文件的组织。
  • Program Header Table: 描述文件中的各种segments，用来告诉系统如何创建进程映像的，即Elf文件如何被操作系统映射到进程的虚拟空间
  • sections 与 segments
    • segments是从装载运行的角度来描述elf文件，sections是从链接的角度来描述elf文件，也就是说，在链接阶段，我们可以忽略program header table来处理此文件，在运行阶段可以忽略section header table来处理此程序（所以很多加固手段删除了section header table）。
    • 从图中我们也可以看出，segments与sections是包含的关系，一个segment包含若干个section（对于相同权限的section合并到一起形成segment）。
    • 由于Elf目标文件不需要被装载，所以目标文件没有程序头表，而Elf可执行文件和共享文件都有程序头表
  • Section Header Table: 包含了文件各个segction的属性信息

  • 该Elf文件头保存在一个**Elf64_Ehdr**结构体中，其结构体具体定义如下：

    该结构体成员变量说明：

    成员 含义
    e_ident[16] e_ident的索引下标及其含义
    名称 值 含义（在上述例子中）
    EI_MAG0 0 文件标识 (0x7f)
    EI_MAG1 1 文件标识 (0x45 ‘E’)
    EI_MAG2 2 文件标识 (0x4c ‘L’)
    EI_MAG3 3 文件标识 (0x46 ‘F’)
    EI_CLASS 4 文件类
    EI_DATA 5 数据编码
    EI_VERSION 6 文件版本
    EI_PAD 7 开始填充字节
    EI_NIDENT 16 e_ident[]的大小

    e_ident[EI_MAG0]至e_ident[EI_MAG3]表示魔数，在上图中，具体含义如下：
    |—含义 值 名称
    |—e_ident[EI_MAG0] 0x7f ELFMAG0
    |—e_ident[EI_MAG1] 0x45 字符E ELFMAG1
    |—e_ident[EI_MAG2] 0x4c 字符L ELFMAG2
    |—e_ident[EI_MAG3] 0x46 字符F ELFMAG3

    e_ident[EI_CLASS]的含义：
    |—含义 值 名称
    |—ELFCLASSNONE 0 无效类
    |—ELFCLASS32 1 32位对象
    |—ELFCLASS64 2 64位对象

    e_ident[EI_DATA]的含义：
    |—含义 值 名称
    |—ELFDATANONE 0 无效编码格式
    |—ELFDATA2LSB 1 小端模式（数据的低位字节放在内存的低地址段）
    |—ELFDATA2MSB 2 大端模式（数据的高位字节放在内存的低地址段）

    e_ident[EI_VERSION] 指定 ELF 头版本号。目前，这个值必须是EV_CURRE NT，正如上文对 e _version 的解释。
    e_ident[EI_PAD] 标记了e_ident未使用字节的开始，这些字节被保留并被设置为0。程序在读取目标文件的时候忽略它们；如果给这些未被使用的字节赋予了含义，则EI_PAD的值改变
    e_type ELF文件类型（每个类型对应一个常量，系统通过这个常量判断ELF真正的文件类型）
    常量 值 说明
    ET_NONE 0 没有文件类型
    ET_REL 1 可重定位文件，一般为.o文件
    ET_EXEC 2 可执行文件
    ET_DYN 3 共享目标文件，一般为.so文件
    ET_CORE 4 核心文件
    ET_LOPROC 0xff00 处理器
    ET_HIPROC 0xffff 处理器
    e_machine Machine
    ELF文件的CPU平台属性
    e_version Version
    ELF版本号，一般为常数1，但是会根据需要改变
    e_entry entry point address
    入口地址：规定ELF程序的入口虚拟地址（系统首先将控制转移到该地址，从而启动进程。如果文件没有关联的入口点，该成员保存为 0。）。操作系统在加载完该程序后从这个地址开始执行进程的指令。可重定位文件一般没有入口地址，则这个地址为0
    e_phoff Start of program headers
    ELF链接视图和执行视图
    e_shoff Start of section headers
    段表在文件中的偏移
    e_flags Flags
    ELF标志位，用来标识一些ELF文件平台相关属性。
    相关常量格式一般为EF_machine_flag，machine为平台，flag为标志
    e_ehsize Size of this headers
    ELF文件头本身大小
    e_phentsize Size of program headers（文件的程序头表中一个条目的字节大小，所有条目大小相同）
    e_phnum Number of program headers（程序头表中条目数量）
    e_shentsize Size of section headers
    段表描述符的大小，一般等于sizeof(ELF32_SHdr)
    e_shnum Number of section headers
    段表描述符数量，值=ELF文件中拥有的段的数量
    e_shstrndx Section header string table index
    段表字符串表所在的段在段表中的下标

3.1.2 通过readelf -S 文件名查看该目标文件的段内容

  • 第一行“There are 14 section hearders, starting at offset 0x428”中0x428 = 4x16^2+2x16+8 = 1064 = Elf文件头中start of section headers的大小

  • 节头表的结构体定义

    该结构体的成员与含义如下：

    成员 含义
    sh_name Section name
    段名 段名是一个字符串，位于一个叫做.shstrtab的字符串。sh_name是段名字符串在 .shstrtab 中的偏移
    sh_type Section type 段的类型 相关常量常以SHT_开头
    常量 值 含义
    SHT_NULL 0 无效段
    SHT_PROGBITS 1 程序段、代码段、数据段都是这种类型的
    SHT_SYMTAB 2 该段内容为符号表
    SHT_STRTAB 3 该段内容为字符串表
    SHT_RELA 4 重定位表，该段包含了重定位信息
    SHT_HASH 5 符号表的哈希表
    SHT_DYNAMIC 6 动态链接信息
    SHT_NOTE 7 提示性信息
    SHT_NOBITS 8 该段在文件中没有内容
    SHT_REL 9 重定位表，该段包含了重定位信息
    SHT_SHLIB 10 保留
    SHT_DNYSYM 11 动态链接的符号表
    sh_flags Section flag 段的标志位，表示该段在进程虚拟地址空间中的属性，相关常量常以SHF_开头
    常量 值 含义
    SHF_WRITE 1 该段在进程空间中可写
    SHF_ALLOC 2 该段在进程空间中必须要分配空间
    SHF_EXECINSTR 4 该段在进程空间中可以被执行
    sh_addr Section Address 段虚拟地址
    如果该地址可以被加载，则sh_addr为该段被加载后在进程地址空间中的虚拟地址；否则sh_addr为0
    sh_offset Section offest 段偏移
    如果该段存在于文件中，则表示该段在文件中的偏移；否则无意义
    sh_size Section size 段的长度
    sh_link与sh_info Section link and Section Infomation 段链接信息。若段的类型与链接相关，则这两部分内容有意义，对于其他类型的段，这两部分内容无意义
    sh_type sh_link sh_info
    SHT_DYNAMIC 该段所使用的字符串表在段表中的下标 0
    SHT_HASH 该段所使用的符号表在段表中的下标 0
    SHT_REL SHT_RELA 该段所使用的相应符号表在段表中的下标 该重定位表所作用的段在段表中的下标 SHT_SYMTAB SHT_DYNSYM 操作系统相关的 操作系统相关的
    other SHN_UNDEF 0
    sh_addralign Section Address Alignment 段地址对齐
    有些段对段地址对齐有要求 由于地址对齐的数量都是2的整数倍，sh_addralign表示的是地址对齐数量中的指数 sh_addr满足：sh_addr%（2 的 sh_addralign 次方）== 0
    sh_entsize Section Entry Size 项的长度
    有些段包含了一些固定大小的项；如果为0，则表示该段不包含固定大小的项

3.1.3 符号表 .symtab

  • 符号表的结构体定义与说明

    该符号表的成员与含义如下：

    成员 含义
    st_name 符号名。该成员包含了该符号名在字符串表中的下标
    st_info 符号类型和绑定信息，该成员低4位表示符号类型，高28位表示绑定信息符号绑定信息
    绑定信息
    宏定义名 值 说明
    STB_LOCAL 0 局部符号，对于目标文件的外部不可见
    STB_GLOBAL 1 全局符号，外部可见
    STB_WEAK 2 弱引用

    符号类型
    宏定义名 值 说明
    STT_NOTYPE 0 未知类型符号
    STT_OBJECT 1 该符号是个数据对象，比如说变量、数组等
    STT_FUNC 2 该符号是个函数或其他可执行代码
    STT_SECTION 3 该符号表示一个段，这种符号必须是
    STB_LOCAL的 STT_FILE 4 该符号表示文件名，一般都是该目标文件所对应的源文件名，他的符号绑定信息一定是STB_LOCAL，并且他的st_shndx（符号所在的段）一定是SHN_ABS
    st_other 该成员目前为0，无用
    st_shndx 符号所在的段。
    符号定义在本目标文件，则表示符号所在的段在段表中的下标。
    符号不是定义在本目标文件中，或者对于部分特殊符号，该值有些特殊，如下：
    宏定义名 值 说明
    SHN_ABS 0xfff1 该符号包含了一个绝对的值
    SHN_COMMON 0xfff2 该符号是一个“COMMON块”类型符号，如未初始化的全局符号定义就是该类型
    SHN_UNDEF 0 该符号未定义。该符号在本目标文件被引用，但是定义在其他目标文件中
    st_value 符号对应的值。这个值跟符号相关，可能是一个绝对值，也可能是一个地址等，不同的符号所对应的值含义不同
    每个符号都有对应的值，若该符号是函数或变量的定义，则符号值就是这个函数或变量的地址，详细如下：
    1. 在目标文件中，如果是符号的定义且该符号不是“COMMON块”类型（即st_shndx不为SHN_COMMON），则该值表示该符号在段中的偏移。即符号所对应的函数或变量位于st_shndx指定的段，偏移st_value的位置
    2.在目标文件中，若符号是“COMMON”类型（sh_shndx为SHN_COMMON）的，则st_value表示该符号的对齐属性
    3.在可执行文件中，st_value表示符号的虚拟地址
    st_size 符号大小。对于包含数据的符号，这个值是该数据类型的大小。

  • 通过命令readelf -s 文件名查看符号表数据

  • 我们可以发现，global_1 global_3 part_3在.data段中，且类型为Object（数据对象)
    global_1 global_2等全局静态变量是全局符号，外部可见
    printf未定义在本文件中，是外部符号

  • 以global_3为例，它的Ndx为3，表明他在段表中下标为3的数据段.data段；Bind为Local表明该数据为局部符号，对于该文件外的其他文件该数据不可见；Type为OBJECT表明该数据为数据对象；Size为4，正好是int类型数据的大小；Value为4，表明该符号位于.data段偏移4的位置

    • 通过命令od -Ax -t x1 -j 172 -N 4 1.o可以查看该符号所代表的数据

3.1.4 重定位表 .rel.表名(暂时没有理解明白)

使用命令readelf -r 文件名查看.o文件中的重定位信息

成员 含义
r_offset 该成员给出应用重定位操作的位置。
对于**==可重定位文件==，该值是段偏移量，即重定位本身描述了怎样修改文件中的另一个节；重定位偏移量指定第二节内的存储单元
对于==可执行文件或共享对象==，该值是受到重定位影响的存储单元的虚拟地址**
r_info 该成员给出了必须要进行重定位的符号表下标 与 要应用的重定位类型
r_addend 该成员指定一个常量加数，用来计算存储在重定位文件中的值
只有ELF32_Rela项含有显式加数，ELF32_Rel项在修改的地方存储了一个非显式的加数

3.2 链接后的Elf文件的文件格式

通过命令gcc 1.o -o 1链接该目标文件

通过上述命令进行链接后，所形成的是可执行目标文件，可以通过ls -al命令查看该文件的详细信息

3.2.1 通过readelf -h 文件名查看链接后的elf文件头

  • 通过与未链接的elf文件头进行比较，发现多了 program header table选项，这是因为可执行文件需要被装载到内存中进行运行，所以需要

  • segment的数量要少于section数量（原因是：一个segment由若干个权限相同的section合并而来）

  • elf文件类型type由REL（可重定位目标文件）变为DYN（共享目标文件）

  • Entry point address相比于未链接前由0x0变为0x1060
    这说明当前程序可由该地址处开始执行

    linux默认程序从_start开始执行（若没有通过链接脚本修改的话）

3.2.3 通过readelf -l 文件名查看程序头表的相关内容

  • 程序头表保存在名为Elf64_Phdr结构体中，其结构体如下： 对于该结构体中各个成员的介绍如下：
    成员 含义
    p_type Segment 的类型
    p_offset Segment 在文件中的偏移
    p_vaddr Segment 的第一个字节在进程虚拟地址空间的起始位置
    p_paddr Segment 的物理装载地址，一般情况下与p_vaddr的值相同
