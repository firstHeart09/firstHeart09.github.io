进程虚存空间分布

第一章 Elf文件的链接视图与执行视图

1.1 背景

我们知道，对于elf文件的解析有两种视图，分别是：链接视图与执行视图。其中链接视图是在链接阶段的视图，可以忽视programe table header（程序头表）；而执行视图是在装载运行时的视图，需要programe table header。当我们对源程序进行编译与汇编后，形成的.o文件，使用命令readelf -h 文件名查看elf文件的文件头，发现当前size of programe header的大小为0，说明此时没有所谓的程序头表，但是当我们将该目标文件链接为可执行文件时，使用上述命令查看文件头，发现size of programe header不为0。

但是当我们将该可执行文件装载到内存中运行时，实际上并不关注每个sections的内容，反而对每个sections的权限十分在意。所以可以采用一种类似于相同权限段合并的方法，将同一权限的段合并到一起，这样合并到一起的段就形成了segment节，这就是programe table header中的每一个segment的由来。

1.2 关于程序头表的介绍

在链接前，没有程序头表一说，即programe table header的大小为0；但是在链接后，程序头表就不为0。

程序头表的相关信息保存在一个名为Elf64_Phdr（64位系统，32位系统类似）的结构体中，该结构体的定义与各参数解释如下：

  • p_type：此字段（4 字节）说明本程序头所描述的段的类型，或者如何解析本程序头的信息。下方为源码中部分常用定义、对应取值及其含义。想要了解其余的定义请参见源码及其对应的注释。

    • /* Legal values for p_type (segment type).  */

#define PT_NULL         0               /* Program header table entry unused */
#define PT_LOAD         1               /* Loadable program segment */
#define PT_DYNAMIC      2               /* Dynamic linking information */
#define PT_INTERP       3               /* Program interpreter */
#define PT_NOTE         4               /* Auxiliary information */
#define PT_SHLIB        5               /* Reserved */
#define PT_PHDR         6               /* Entry for header table itself */
#define PT_TLS          7               /* Thread-local storage segment */
#define PT_NUM          8               /* Number of defined types */
#define PT_LOOS         0x60000000      /* Start of OS-specific */
#define PT_GNU_EH_FRAME 0x6474e550      /* GCC .eh_frame_hdr segment */
#define PT_GNU_STACK    0x6474e551      /* Indicates stack executability */
#define PT_GNU_RELRO    0x6474e552      /* Read-only after relocation */
#define PT_GNU_PROPERTY 0x6474e553      /* GNU property */
#define PT_LOSUNW       0x6ffffffa
#define PT_SUNWBSS      0x6ffffffa      /* Sun Specific segment */
#define PT_SUNWSTACK    0x6ffffffb      /* Stack segment */
#define PT_HISUNW       0x6fffffff
#define PT_HIOS         0x6fffffff      /* End of OS-specific */
#define PT_LOPROC       0x70000000      /* Start of processor-specific */
#define PT_HIPROC       0x7fffffff      /* End of processor-specific */

    • 0x1 可执行 0x2 可写 0x4 可读

    • PF_MASKOS 所覆盖的权限值是为特殊操作系统保留的

    • PF_MASKPROC 所覆盖的权限值是为特殊处理器保留的。

  • p_offset：此字段（8 字节）给出本段内容在文件中的位置，即段内容的开始位置相对于文件开头的偏移量。

  • p_vaddr：此字段（8 字节）给出本段内容的开始位置在进程空间中的虚拟地址。

  • p_paddr：此字段（8 字节）给出本段内容的开始位置在进程空间中的物理地址。对于目前大多数现代操作系统而言，应用程序中段的物理地址事先是不可知的，所以目前这个 成员多数情况下保留不用，或者被操作系统改作它用。

  • p_filesz：此字段（8 字节）给出本段内容在文件中的大小，单位是字节，可以是 0。

  • p_memsz：此字段（8 字节）给出本段内容在内存中的大小，单位是字节，可以是 0。

  • p_align：此字段（8 字节）指明本段内容如何在内存和文件中对齐。如果该值为 0 或 1，表明没有对齐要求；否则，p_align 应该是一个正整数，并且是 2 的幂次数。p_vaddr 和 p_offset 在对 p_align 取模后应该相等。注：对于可装载的段来说，其 p_vaddr 和 p_offset 的值至少要向内存页面大小对齐。

使用命令readelf -l 文件名查看程序头

  • Elf file type is DYN (Position-Independent Executable file)：说明此时elf文件的类型为DYN
  • Entry point 0x106表明当前入口地址为0x106
  • INTERP在LOAD之前（正如p_type中PT_INTERP中说的那样，该segment在每个elf文件中至多出现一次，且在LOAD类型的SEGMENT的前面）
  • Section to Segment mapping说明的是每个Segment由哪几个sections组成以及排序方式
  • 每个LOAD类型的segment的大小计算方式：从Section to Segment mapping中我们可以看到02号（02代表程序头表中的第3个segment）由11个sections组成，分别是.interp .note.gnu.property .note.gnu.build-id .note.ABI-tag .gnu.hash .dynsym .dynstr .gnu.version .gnu.version_r .rela.dyn .rela.plt
  • 通过readelf -S 文件名中.rela.plt段的Address为0x610，size的大小为0x18，则0x610 + 0x18 = 0x628，所以第一个LOAD的的filesize为0x628。且对齐方式为0x1000（因为每个页默认大小为0x1000即4096）
  • 第二个LOAD segment从.init sections开始到.fini sections结束。.init的Address为0x1000，.fini的Address为0x1188，size为0xd，则0xd + 0x1188 - 0x1000 = 0x195即为第二个LOAD的size
